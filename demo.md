## Project Architecture Summary

### **Overall Architecture**

The **kaseya-demo** project is a comprehensive O365 security event processing and AI triage system with two major components:

1. **O365-events-simulator** (Go): Event generation and routing pipeline
2. **triage-agent** (TypeScript): LLM-powered security triage agent

The system uses **Apache Kafka** as the event backbone and **LangGraph** for agentic AI reasoning, with the entire stack deployable to **Kubernetes (K3s)** via Helm charts.

---

### **1. Architecture Overview**

```
O365 Event Sources
        ↓
   [Kafka Producer] (Go)
        ↓
   [3 Input Topics: topic-1, topic-2, topic-3]
        ↓
   [Consumer Service] (Go) → Routes scenarios → topic-false-positive
        ↓
   [Triage Agent] (TypeScript) → Consumes false positives
        ↓
   [AI Agent] (LangGraph + OpenAI/LM Studio) → Tool calling
        ↓
   [topic-triage-verdict] → Monitoring dashboard
```

### **2. Microsoft O365/Azure Events Handled**

The system simulates and processes **15 distinct O365 security alert categories**. These are generated by the Go producer and include:

#### **False Positive Scenarios** (benign explanations):
1. **ImpossibleTravel** - Two sign-ins from distant geos in short time (FP: VPN exit nodes, corporate proxies)
2. **BulkSharePointDownload** - Large volume file downloads (FP: OneDrive sync clients, authorized migrations)
3. **MFADisabledForUser** - MFA removed from account (FP: IT helpdesk break-glass procedures)
4. **GuestAccountAdded** - External user invited (FP: vendor/partner onboarding, KPMG contractors)

#### **True Positive Scenarios** (actual threats):
5. **ExternalForwardingRuleCreated** - Inbox rule forwards to external domain (TP: Email exfiltration, BEC persistence)
6. **OAuthAppConsentGrant** - Third-party app granted broad permissions (TP: unverified apps requesting Mail.ReadWrite, Directory.ReadWrite)
7. **PasswordSpray** - Multiple failed logins across accounts (TP: credential attacks from known-bad IPs)
8. **SuspiciousInboxRule** - Rule hides/deletes/redirects emails (TP: Business Email Compromise hiding finance emails)
9. **AnonymousIPSignIn** - Login from Tor/anonymous proxy (TP: threat actor anonymization)

#### **Ambiguous Scenarios** (mixed indicators):
10. **SuspiciousSignIn** - Unfamiliar location/device, legacy auth (risky but can be business travel)
11. **MassMailboxDeletion** - High-volume deletes in short window (could be inbox cleanup or ransomware)
12. **RiskyUserDetected** - Azure AD Identity Protection flag (legitimate after password reset or travel)
13. **MailboxPermissionGranted** - Full Access/Send-As granted (legitimate for shared mailbox, but can enable lateral movement)
14. **AdminConsentGranted** - Admin granted app permissions tenant-wide (legitimate IT rollout or supply chain attack)
15. **DataExfiltrationAlert** - Defender detected potential exfil (DLP test vs. real data theft)

**Event types NOT triaged** (background events, passed through to OpenSearch):
- AADSignin, ActivityLog, SecurityAlert, Diagnostic, OutlookMail, OutlookCalendar, OutlookAdmin

### **3. AI Triage Functionality**

#### **Triage Agent Architecture** (LangGraph State Machine)

The agent uses a **3-node agentic loop**:

```
START → [Agent Node] → (invoke model with tools) → [Tools Node] → (execute tools)
         ↑                                              ↓
         └──────────────────────────────────────────┘
                      (feedback loop)
         
         → [ParseVerdict Node] → END
```

**Agent System Prompt Key Instructions:**
- Investigates alerts using **evidence collection methodology**
- Extracts key indicators: UPN, ClientIP, Operation, Workload, ObjectId
- Verdict thresholds: ≥70% confidence → conclusive (FP/TP), <70% → INCONCLUSIVE

---

#### **AI Triage Tools** (5 Tool-Use Capabilities)

1. **check_o365_user_context** - Microsoft Graph API queries:
   - `sign_in_history`: Recent logins, locations, failed attempts
   - `mfa_status`: MFA enabled/disabled, authentication methods
   - `mailbox_rules`: Forwarding/deletion rules, suspicious patterns
   - `delegated_permissions`: Full mailbox access grants
   - `oauth_consents`: Third-party app permissions
   - `recent_admin_actions`: User's privileged actions
   - `risky_sign_ins`: Azure AD Identity Protection detections

2. **lookup_threat_intel** - Threat intelligence lookups:
   - Real integration: AbuseIPDB, VirusTotal
   - Simulated fallback: Known-bad IPs (198.51.100.0/24), known malicious domains
   - Returns verdicts: MALICIOUS (score >50), SUSPICIOUS (>10), CLEAN (<10)

3. **query_siem_events** - SIEM/Elasticsearch queries:
   - Filters by source IP, hostname, user ID, time range
   - Returns recent security events for correlation
   - Detects broader attack patterns, login failures, lateral movement

4. **analyze_network_traffic** - Network behavior analysis:
   - Flow summary, DNS queries, firewall hits, bandwidth anomalies
   - Detects DGA, suspicious destinations, unusual traffic patterns
   - Verifies IP reputation and geolocation data

5. **check_endpoint_status** - EDR agent telemetry:
   - Process activity, network connections, file activity
   - Detects suspicious processes, anomalous connections, encryption activity
   - Overall risk scoring and policy compliance

**Fallback Behavior:** All tools include **simulated data generation** for demo mode (no external APIs required).

### **4. Monitoring & Dashboard Components**

#### **Verdict Monitor (TypeScript/Kafka Consumer)**
- **File:** `/triage-agent/src/monitor.ts`
- **Run Command:** `npm run monitor`
- **Purpose:** Real-time display of triage verdicts from `topic-triage-verdict`
- **Features:**
  - Color-coded verdict badges (RED=TRUE_POSITIVE, GREEN=FALSE_POSITIVE, YELLOW=INCONCLUSIVE)
  - Confidence score bar graph (█████░░░░ 75%)
  - Evidence trail display with source attribution
  - Timestamp, triage ID, alert ID, reasoning
  - Tool call count tracking (measures agent iteration complexity)

#### **OpenSearch Integration** (Consumer Service)
- **File:** `/O365-events-simulator/cmd/consumer/main.go`
- **Functionality:**
  - Indexes all O365 events into OpenSearch/Elasticsearch
  - Filters scenario events → forwards to `topic-false-positive`
  - Provides searchable event history for incident response
  - Index: `o365-events` with fields: eventType, workload, userId, clientIp, location, resultStatus

#### **OpenSearch Dashboards** (Helm-deployed)
- Visualize event streams across all 3 topics
- Time-series analysis of security alerts
- User behavior analytics, geographic heatmaps

```

Consumer reads → identifies scenario event → forwards to topic-false-positive
  (also indexes to OpenSearch for history)

         ↓

  - Normalizes raw event to FalsePositiveEvent
  - Extracts: userId=bob.smith@contoso.com, clientIP=198.51.100.77, op=New-InboxRule
  - Creates triage prompt with event details

         ↓

  1. Agent node: "Is this email forwarding to attacker-controlled account?"
     → Tool call: check_o365_user_context(bob.smith@contoso.com, delegated_permissions)
     → Result: "No prior delegation, mailbox normal"
     
  2. Agent node: "Is the source IP known malicious?"
     → Tool call: lookup_threat_intel(198.51.100.77, ip)
     → Result: "MALICIOUS, abuse score 85"
```

### **8. Configuration (Environment Variables)**

**Triage Agent:**
- `KAFKA_BROKERS`: Kafka broker addresses (default: 192.168.0.103:9092)
- `KAFKA_INPUT_TOPIC`: Subscribe topic (default: topic-false-positive)
- `KAFKA_OUTPUT_TOPIC`: Publish verdicts to (default: topic-triage-verdict)
- `LM_STUDIO_BASE_URL`: LLM API endpoint (default: http://localhost:1234/v1)
- `AGENT_MODEL`: Model name (default: google/gemma-4-e4b)
- `AGENT_MAX_ITERATIONS`: Recursion limit (default: 15)
- `GRAPH_API_TOKEN`: Microsoft Graph Bearer token (optional, uses simulation if absent)
- `ABUSEIPDB_API_KEY`, `VIRUSTOTAL_API_KEY`: Threat intel keys (optional)
- `SIEM_API_URL`, `SIEM_API_KEY`: Elasticsearch/OpenSearch access (optional)
- `EDR_API_URL`, `EDR_API_KEY`: EDR platform access (optional)
- `LOG_LEVEL`: Winston log level (default: info)

**O365 Simulator:**
- `KAFKA_BROKER`: Broker address (default: fedora:9092)
- `OPENSEARCH_URL`: OpenSearch endpoint (default: http://opensearch-cluster-master.opensearch.svc.cluster.local:9200)

```
152	function parseVerdictNode(state: AgentState): Partial<AgentState> {
153	  // Find the last AI message with text content
154	  let finalText = "";
155	  for (let i = state.messages.length - 1; i >= 0; i--) {
156	    const msg = state.messages[i];
157	    if (msg._getType() !== "ai") continue;
158	    const content = msg.content;

161	      break;
162	    }
163	    if (Array.isArray(content)) {
164	      const text = content
165	        .filter((c): c is { type: string; text: string } => typeof c === "object" && c.type === "text")
166	        .map((c) => c.text)
167	        .join("");
168	      if (text.trim()) { finalText = text; break; }
169	    }
170	  }
171	
172	  const jsonMatch = finalText.match(/```json\s*([\s\S]*?)```\s*$/);
173	  if (jsonMatch) {

176	        verdict: Verdict;

196	      ? "TRUE_POSITIVE"

198	      ? "FALSE_POSITIVE"
```

## Slide 1: The Problem

Security teams are drowning in alerts. Microsoft 365 environments generate thousands of security events daily — but **95%+ are false positives**. Analysts spend hours investigating benign activity, leaving real threats unaddressed.

**The challenge:** How do you automatically distinguish:
- A VPN exit node from a compromised account?
- An IT helpdesk action from an attacker disabling MFA?
- An OneDrive sync from data exfiltration?

## Slide 4: AI Triage — How the Agent Thinks

The agent uses a **LangGraph agentic loop** — it reasons iteratively, calling tools until it has enough evidence for a confident verdict.

```
┌─────────────────────────────────────────┐
│             Agent Node                  │
│  "What evidence do I need next?"        │
│  → Selects tool, forms hypothesis       │
└──────────────┬──────────────────────────┘
               │ tool call
               ▼
┌─────────────────────────────────────────┐
│             Tools Node                  │
│  Executes: Graph API / ThreatIntel /    │
│  SIEM / Network / EDR                   │
└──────────────┬──────────────────────────┘
               │ results
               ▼
        (loop until confident)
               │
               ▼
┌─────────────────────────────────────────┐
│         Parse Verdict Node              │
│  Extracts structured JSON verdict       │
│  Verdict: FALSE_POSITIVE / TRUE_POSITIVE│
│           / INCONCLUSIVE                │
│  Confidence: ≥70% → conclusive          │
└─────────────────────────────────────────┘
```

### The 5 Triage Tools

| Tool | What It Queries | Key Signals |
|---|---|---|
| **check_o365_user_context** | Microsoft Graph API | Sign-in history, MFA status, mailbox rules, delegated permissions, OAuth consents, recent admin actions |
| **lookup_threat_intel** | AbuseIPDB + VirusTotal | IP reputation score, malicious domain lookups, known-bad ranges |
| **query_siem_events** | OpenSearch / Elasticsearch | Correlated events, login failures, lateral movement patterns |
| **analyze_network_traffic** | EDR / network flows | DNS queries, DGA detection, bandwidth anomalies, firewall hits |
| **check_endpoint_status** | EDR telemetry | Process activity, suspicious connections, file encryption activity |

## Slide 7: Deployment — Kubernetes on K3s

```

  ├── producer (Go, 1 replica) — generates events continuously
  ├── consumer (Go, 1 replica) — routes + indexes events
  └── triage-agent (Node.js, 1 replica) — AI verdicts

Namespace: kafka
  └── Kafka broker (topic-1, topic-2, topic-3, topic-false-positive, topic-triage-verdict)

Namespace: opensearch
  └── OpenSearch cluster — event store + SIEM correlation
```
